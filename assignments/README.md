## 设计理念

### 防 AI 速通

每次作业都包含：

1. **Edge 测试**：AI 容易忽略的边界情况
2. **反思报告**：要求展示 AI 交互和改进过程
3. **迭代记录**：要求诚实记录 AI 初版的问题

### 真实场景驱动

- 作业 1：个人信息处理 → 实用的格式化输出
- 作业 2：成绩分析 → 真实的数据处理需求
- 作业 3：文件整理 → 日常自动化脚本
- 作业 4：数据可视化 → 数据分析报告
- 作业 5：完整项目 → 端到端的软件开发

### 能力递进

```
基础语法 → 函数设计 → 文件处理 → 数据分析 → 综合应用
    ↓         ↓          ↓          ↓          ↓
  作业1     作业2      作业3      作业4      作业5
```

## 通用要求

### 提交方式

- 所有作业通过 Gitea 仓库提交
- commit message 包含 `完成作业` 触发自动评分
- 或推送 `submit` 标签触发

### 目录结构

```
assignment-XX/
├── .gitea/workflows/autograde.yml
├── src/                    # 源代码
├── tests/                  # 测试（学生可见部分）
├── README.md               # 作业说明
├── REPORT.md               # 反思报告
└── requirements.txt        # 依赖
```

### REPORT.md 通用要求

每份反思报告必须包含：

1. **AI 交互记录**：展示关键 Prompt 和对话
2. **问题发现**：AI 代码中发现的问题
3. **改进过程**：你的修改和原因

## LLM 评分规则

### 评分维度

| 维度 | 权重 | 说明 |
|-----|------|------|
| AI 交互过程 | 40% | 展示真实的 Prompt 和迭代 |
| 问题发现 | 30% | 识别 AI 代码的不足 |
| 改进说明 | 30% | 修改合理，有解释 |

### 等级标准

| 等级 | 标准 |
|-----|------|
| A | 完整展示迭代，发现多个问题，改进有理有据 |
| B | 有迭代记录，发现 1-2 个问题 |
| C | 仅基本完成，对 AI 代码无审查 |
| D | 无反思内容或明显虚构 |

---

## 边界处理

### 分数复核

- 及格线附近自动标记 `need_review`
- LLM 置信度 < 0.7 送人工复核

### 异常标记

- `empty_answer`：答案为空
- `off_topic`：答非所问
- `suspected_plagiarism`：疑似抄袭
